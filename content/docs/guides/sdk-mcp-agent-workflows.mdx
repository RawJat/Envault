This guide covers Envault's TypeScript SDK and MCP server for machine-agent workflows with a strict human-in-the-loop (HITL) approval model.

## Packages

- SDK package: `@dinanathdash/envault-sdk`
- MCP package: `@dinanathdash/envault-mcp-server`

## Install and Update

### SDK

Install SDK:

```bash
envault sdk install
```

Update SDK to latest:

```bash
envault sdk update
# npm fallback:
npm install @dinanathdash/envault-sdk@latest
```

### MCP

Our MCP server integrates natively with tools like Claude Desktop, Cursor, and VS Code (RooCode/Cline). To install and configure the integration, simply run the setup command from the Envault CLI.

Install MCP strictly for your current workspace (.vscode):

```bash
envault mcp install --local
```

Or install it globally for all your IDEs and Claude Desktop:

```bash
envault mcp install
```

Update MCP integration to the latest runtime behavior:

```bash
envault mcp update
# npm fallback (standalone package installs):
npm install -g @dinanathdash/envault-mcp-server@latest
```

When you use this command, the CLI completely automates creating the standard `mcp.json` or `cline_mcp_settings.json` format configuration for you pointing to `npx -y @dinanathdash/envault-mcp-server@latest`.

## HITL Approval Flow

1. Agent requests a mutation through SDK or MCP.
2. Envault creates a `pending_approvals` record and returns `202` with `approval_id` and `approval_url`.
3. Human approves from web UI or terminal:

```bash
envault approve <approval_id>
```

4. Agent polls approval status and resumes only if approved.
5. Approval payload is one-time consumable (burn-after-read behavior on approved retrieval).

<Callout type="info">
  Approval events are written to audit logs as `AGENT_APPROVED_CHANGE` or `AGENT_REJECTED_CHANGE`.
</Callout>

## API Surfaces

- `POST /api/sdk/auth/delegate`: mints scoped delegated agent token (`envault_agt_...`)
- `POST /api/sdk/secrets`: creates pending approval for mutation payloads
- `GET /api/sdk/approvals/:id/status`: polling endpoint for pending/approved/rejected/expired states
- `POST /api/approve/:id`: approve or reject mutation execution
- `GET /api/sdk-version`: SDK version compatibility (`latest_version`, `min_supported_version`)

## Access Controls and Kill Switches

Agent mutations require all of the following:

- Valid delegated agent token and project scope
- Global user-level agent access enabled
- Project-level agent access enabled
- Human approval for mutation execution

Global and project kill switches can be toggled in the UI under security and project settings.

## MCP Tools

The MCP server exposes these tools:

- `envault_status`
- `envault_pull`
- `envault_push`
- `envault_approve`
- `envault_set_local_key`
- `envault_remove_local_key`

## Release Streams

SDK and MCP are versioned independently using dedicated release workflows:

- SDK tags: `sdk-v<version>`
- MCP tags: `mcp-v<version>`

This keeps package updates decoupled from CLI and web releases.

## AI Context Safety (`.copilotignore`)

To prevent your raw `.env` secrets from being inadvertently read and uploaded into LLM context windows (via tools like GitHub Copilot or Cursor), Envault automatically guards your pulled environment variables.
Whenever you run `envault pull` or initialize a project, Envault automatically updates `.gitignore` and `.copilotignore` to exclude `.env` files.

## Windows and GUI Integrations

If you integrate the MCP server into editors like VS Code or Claude Desktop, you might historically encounter `spawn npx ENOENT` errors on Windows or macOS GUI environments where the `PATH` isn't completely passed down.
Installing via `envault mcp install` now automatically hooks the configurations to use cross-platform safe executions (`npx -y`), eliminating startup crashes.
