This guide covers Envault's TypeScript SDK and MCP server for machine-agent workflows with a strict human-in-the-loop (HITL) approval model.

## Packages

- SDK package: `@dinanathdash/envault-sdk`
- MCP package: `@dinanathdash/envault-mcp-server`

## Install and Update

### SDK

Install SDK:

```bash
envault sdk install
```

Update SDK to latest:

```bash
envault sdk update
```

npm fallback:

```bash
npm install @dinanathdash/envault-sdk@latest
```

Global npm install/update mode:

```bash
envault sdk update --global
```

### MCP

Our MCP server integrates with Claude Desktop, Cursor, and VS Code (RooCode/Cline).

Primary install path (recommended): copy-paste raw MCP JSON config. This does not require installing the Envault CLI.

1. Generate an `ENVAULT_TOKEN` from the Envault web dashboard (Account/Settings -> MCP Token).
2. Copy the MCP config below into your MCP client settings file.
3. Replace `<YOUR_ENVAULT_TOKEN>` before saving.

Claude Desktop / Cursor format (`mcpServers`):

```json
{
  "mcpServers": {
    "envault": {
      "command": "npx",
      "args": ["-y", "@dinanathdash/envault-mcp-server@latest"],
      "env": {
        "ENVAULT_TOKEN": "<YOUR_ENVAULT_TOKEN>",
        "ENVAULT_BASE_URL": "https://www.envault.tech"
      }
    }
  }
}
```

VS Code MCP format (`servers`):

```json
{
  "servers": {
    "envault": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@dinanathdash/envault-mcp-server@latest"],
      "env": {
        "ENVAULT_TOKEN": "<YOUR_ENVAULT_TOKEN>",
        "ENVAULT_BASE_URL": "https://www.envault.tech"
      }
    }
  },
  "inputs": []
}
```

### MCP config file locations by platform

- macOS (Claude Desktop): `~/Library/Application Support/Claude/claude_desktop_config.json`
- Windows (Claude Desktop): `%APPDATA%\\Claude\\claude_desktop_config.json`
- Linux (Claude Desktop): `~/.config/Claude/claude_desktop_config.json`
- Cursor: open Cursor Settings -> MCP and paste the same `mcpServers.envault` block in the MCP JSON editor.
- VS Code MCP clients (for example Cline/RooCode): use the extension MCP settings file (commonly `.vscode/mcp.json` in the current workspace).

If Windows cannot resolve `npx` from GUI-launched apps, use `npx.cmd` (or absolute path):

```json
{
  "servers": {
    "envault": {
      "type": "stdio",
      "command": "npx.cmd",
      "args": ["-y", "@dinanathdash/envault-mcp-server@latest"],
      "env": {
        "ENVAULT_TOKEN": "<YOUR_ENVAULT_TOKEN>",
        "ENVAULT_BASE_URL": "https://www.envault.tech"
      }
    }
  },
  "inputs": []
}
```

<Callout type="info">
  `ENVAULT_TOKEN` for MCP is generated directly from the Envault web dashboard. CLI installation is optional for MCP-only workflows.
</Callout>

<Callout type="info">
  Standalone MCP coverage (no local Envault CLI required): `envault_status`, `envault_context`, `envault_pull`, `envault_push`, `envault_deploy`, `envault_approve`, `envault_diff`, and `autoPush` from local key helpers. Other maintenance/setup commands remain CLI-backed.
</Callout>

<Callout type="warn">
  If standalone MCP returns `401 Unauthorized`, the token is invalid for the target API host. Common causes: masked/partial token copy, token expired, token revoked/regenerated, or host mismatch (`ENVAULT_BASE_URL` points to a different Envault instance than where the token was created).

  For Envault Cloud, set `ENVAULT_BASE_URL` to `https://www.envault.tech` (canonical host). Some clients can drop `Authorization` headers across cross-host redirects (for example `envault.tech` -> `www.envault.tech`).

  After changing token or env values in MCP config, fully restart the MCP server (or reload the IDE window) so the new environment is applied.
</Callout>

<Callout type="warn">
  MCP dashboard token behavior:
  - Only one active MCP token per user account at a time.
  - Token is shown in full only once at generation time; after closing the dialog, the dashboard shows a masked value.
  - Token TTL options are fixed to 7, 15, or 30 days and is automatically invalidated/deleted after expiry.
  - Expiry is stored as end-of-day UTC so scheduled cleanup can run daily with low database overhead.
  - Users can regenerate within active validity or create a new token after expiry.
  - Token format stays compatible with current Envault auth parsing used by MCP/SDK.
</Callout>

### MCP token cleanup automation

MCP web tokens are auto-cleaned through a protected cron endpoint:

- Endpoint: `/api/cron/mcp-token-cleanup`
- Auth: `Authorization: Bearer <CRON_SECRET>`
- Schedule: daily (recommended for free-tier efficiency)

Set `CRON_SECRET` in your runtime environment (local `.env.local` and production environment variables), then schedule the job in Supabase SQL Editor using:

- `supabase/migrations/20260405000000_add_mcp_web_token_columns_to_profiles.sql`
- `scripts/setup-mcp-token-cleanup-cron.sql`

<Callout type="info">
  The cron SQL file unschedules legacy hourly cleanup names before creating the daily schedule. This is intentional and safe to run repeatedly.
</Callout>

### Local validation checklist (before deploy)

1. Generate MCP token from Settings -> Security.
2. Validate token on CLI-protected routes:

```bash
curl -k -i \
  -H "Authorization: Bearer <FULL_MCP_TOKEN>" \
  "https://envault.localhost:1355/api/cli/me"
```

3. Force-expire token in database, then trigger cleanup manually:

```bash
curl -k -i \
  -H "Authorization: Bearer <CRON_SECRET>" \
  "https://envault.localhost:1355/api/cron/mcp-token-cleanup"
```

4. Confirm token is invalid after cleanup and MCP profile token fields are cleared.


npm fallback (standalone package installs):

```bash

```

## HITL Approval Flow

1. Agent requests a mutation through SDK or MCP.
2. Envault creates a `pending_approvals` record and returns `202` with `approval_id` and `approval_url`.
3. Human approves from web UI or terminal:

```bash
envault approve <approval_id>
```

4. Agent polls approval status and resumes only if approved.
5. Approval payload is one-time consumable (burn-after-read behavior on approved retrieval).

<Callout type="info">
  Approval events are written to audit logs as `AGENT_APPROVED_CHANGE` or `AGENT_REJECTED_CHANGE`.
</Callout>

## API Surfaces

- `POST /api/sdk/auth/delegate`: mints scoped delegated agent token (`envault_agt_...`)
- `POST /api/sdk/secrets`: creates pending approval for mutation payloads
- `GET /api/sdk/approvals/:id/status`: polling endpoint for pending/approved/rejected/expired states
- `POST /api/approve/:id`: approve or reject mutation execution
- `GET /api/sdk-version`: SDK version compatibility (`latest_version`, `min_supported_version`)

## Access Controls and Kill Switches

Agent mutations require all of the following:

- Valid delegated agent token and project scope
- Global user-level agent access enabled
- Project-level agent access enabled
- Human approval for mutation execution

Global and project kill switches can be toggled in the UI under security and project settings.

## Release Streams

SDK and MCP are versioned independently using dedicated release workflows:

- SDK tags: `sdk-v<version>`
- MCP tags: `mcp-v<version>`

This keeps package updates decoupled from CLI and web releases.

## AI Context Safety (`.copilotignore`)

To prevent your raw `.env` secrets from being inadvertently read and uploaded into LLM context windows (via tools like GitHub Copilot or Cursor), Envault automatically guards your pulled environment variables.
Whenever you run `envault pull` or initialize a project, Envault automatically updates `.gitignore` and `.copilotignore` to exclude `.env` files.

## Windows and GUI Integrations

If you integrate the MCP server into editors like VS Code or Claude Desktop, you might historically encounter `spawn npx ENOENT` errors on Windows or macOS GUI environments where the `PATH` isn't completely passed down.
Installing via `envault mcp install` now automatically hooks the configurations to use cross-platform safe executions (`npx -y`), eliminating startup crashes.
